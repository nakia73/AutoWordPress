## 4. 技術的破綻リスク

### 4.1 致命的リスク（Critical）

1. **単一VPS構成でのSPOF**
   - 影響: 全ユーザーのサービス停止
   - 発生確率: 中（ハードウェア障害、DDoS攻撃）
   - 対策案: Phase 1からRedundancy設計を検討

2. **WordPress REST API認証トークン漏洩**
   - 影響: 任意記事の改ざん・削除
   - 発生確率: 低〜中
   - 対策案: トークンローテーション、IP制限必須化

3. **LLMコスト爆発**
   - 影響: 月額コストが収益を上回る
   - 発生確率: 高（価格設定次第）
   - 対策案: 記事生成上限の厳格化、コスト監視

### 4.2 高リスク（High）

1. **Multisite性能劣化**
   - 500サイト以上でクエリパフォーマンス低下
   - 対策案: HyperDBやQuery Cacheの早期検討

2. **Vercel Serverless Timeout**
   - 300秒制限で長時間ジョブが中断
   - 対策案: 確実なWorker分離アーキテクチャ

## 13. アーキテクチャの根本的矛盾（Iteration 3追加）

### 13.1 ファーストプリンシプルからの検証

**本質的な問い:** 「なぜWordPressなのか？」

| 理由（ドキュメント記載） | 反論 | 結論 |
|------------------------|------|------|
| Exit Strategy（移管可能性） | 記事データだけならJSON/Markdownでも可。WPテーマ・プラグインの移管は複雑 | 一部妥当 |
| SEO最適化済み | Headless CMS + Next.js SSGでも同等以上のSEO可能 | 疑問 |
| ユーザー馴染み | ターゲット「個人開発者」はWPより技術寄りツールに馴染む | 疑問 |

### 13.2 アーキテクチャ複雑性の問題

| 問題ID | 問題 | 詳細 | 深刻度 |
|--------|------|------|--------|
| ARCH-001 | 3層の異種環境 | Vercel (Node.js) + VPS (PHP/WordPress) + Cloudflare。障害切り分け困難 | 高 |
| ARCH-002 | 2つのDB運用 | PostgreSQL (Neon) + MariaDB (WordPress)。データ整合性・運用コスト2倍 | 高 |
| ARCH-003 | 認証の二重管理 | NextAuth (アプリ) + WordPress Auth (WP)。Phase 9 SSOで複雑化 | 高 |
| ARCH-004 | API通信の脆弱性 | VPS ↔ Vercel間の通信がシークレット依存。ネットワーク隔離なし | 中 |

### 13.3 代替アーキテクチャの未検討

| 代替案 | メリット | デメリット | 検討有無 |
|--------|---------|-----------|---------|

| Ghost CMS | モダン、API First | 移管性の説明が難しい | 未検討 |
| 独自CMS | 完全制御 | 開発コスト大 | 未検討 |
| Notion API + Super.so | 開発コスト最小 | 機能制限 | 未検討 |

### 14.1 フェーズ順序の矛盾

### 14.2 MVP定義の再検証

**ファーストプリンシプル:** 「最小限で価値を証明するには何が必要か？」

| 現MVP定義 | 問題点 |
|-----------|--------|

| 10社限定Beta | 統計的に有意なフィードバックを得られない |

**提案: MVPの再定義**

- または「自動化」の訴求を控え、「AI記事生成ツール」としてMVP

### 25.2 ドキュメント設計の問題

| 問題 | 影響 |
|------|------|
| 「What」が先、「Why/Who」が後 | 技術選定が目的化し、顧客価値が後付け |
| Phase設計が技術駆動 | 「何を作るか」が先、「誰のために」が不明確 |
| 成功基準が技術指標中心 | 「10分で生成」等、顧客価値との紐づけが弱い |

|------|---------|

| 複雑な3層構成 | シンプルに始める発想の欠如 |
| 2種類のDB運用 | 技術的整合性より「使いたい技術」を優先 |
