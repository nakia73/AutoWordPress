## 回答済み項目

### A. コアコンセプト

#### A1. 核心的な価値提案

**決定事項:**

このサービスの最も革新的な価値提案は、**技術的な新奇性ではなく「パッケージング」**にある。

> AIエージェントツールを組み込むことによって、「こんなに簡単に自分のWordPressブログを作成し、勝手に運用してくれる環境を手に入れることができる」というパッケージング。

**背景・理由:**

1. **統合された体験の不在**
   - 今までは：
     - WordPressは自分で立ち上げる必要があった
     - サーバーを自分で借りる必要があった
     - 記事自動生成の拡張機能は存在したが、バラバラだった
   - 本サービスは：
     - 作りたいコンセプトまたは独自ドメインを入力し、簡単な設定をするだけ
     - WordPressセットアップ → 記事スケジューリング → 自動生成 → 自動投稿 を**一気通貫で実行**

2. **時代背景との適合**
   - 2026年、バイブコーディング（Vibe Coding）が爆発的に普及
   - 大きな問題：「素晴らしいプロダクトが完成したのに、誰にも見てもらえない」
   - アプリは作れるようになったが、マーケティングはわからない人が大量に存在
   - そのような人たちに「AIツールで簡単にブログ生成・集客できる」ことは非常に魅力的

3. **見せ方の重要性**
   - 技術的に優れているかどうかではなく、**見せ方として魅力的に映る**ことが重要
   - バイブコーディングの潮流に乗った「バズりやすい切り口」として訴求

4. **機能的価値のみを訴求しない**
   - 「WordPressを自動化できます」という**汎用的・機能的な価値訴求ではない**
   - 「バイブコーディングで作ったプロダクトの集客を自動化できる」という**文脈に根ざした体験価値**を提供
   - ターゲットの課題（マーケティング手段がない、SEOがわからない）に直接響く訴求

#### A3. ワンラインピッチ（10秒説明）

**決定事項:**

> **「作ったアプリ、誰にも見つけてもらえない？ブログ集客をAIが全自動で。」**

**選定理由:**
- ペイン（作ったけど使われない）を直接突く
- ターゲット（バイブコーダー）が「自分のことだ」と即座に認識できる
- 解決策（AIブログ集客）も明確に提示

#### A5. 原体験（このサービスを作ろうと思ったきっかけ）

**決定事項:**

創業者自身の実体験と市場観察に基づく：

1. **自身の経験**
   - バイブコーディングでプロダクトを作ったが、集客に困った
   - 作ることはできても、届けることの難しさを痛感

2. **周囲の観察**
   - 同じ課題を抱えるバイブコーダーが多数存在することを確認
   - 「作ったけど使われない」問題は自分だけではなかった

3. **市場機会の認識**
   - この課題を解決するツールが存在しないことに気づいた
   - バイブコーディングの爆発的普及というタイミングでの参入を決断

**ストーリーテリング用サマリー:**
> 「自分でアプリを作れるようになった。でも誰にも使ってもらえない。同じ悩みを持つ人がたくさんいた。だから作った。」

#### A6. 5年後のビジョン

**決定事項:**

> **AI時代の集客チャネルにおけるデファクトスタンダードを目指す**

---

**将来のブログ運用の姿（予測）:**

次の時代には、以下の2つが**両立して運用されるのがデファクトスタンダード**になる：

| 担当 | 領域 | 理由 |
|------|------|------|
| **AI** | 一般的な知識・情報のまとめ | 人間を超えるクオリティが発揮できる。人間がやるのは非効率。 |
| **人間** | 独自体験・独自の感性・オリジナルの視点 | AIでは再現不可能。必ず残る領域。 |

---

**目指すべき姿（使命）:**

1. **量産ツールではない**
   - ただ記事を量産できる自動ツールを目指すのではない
   - 「たくさん作れる」ではなく「良いものを作れる」

2. **クオリティへのフォーカス**
   - 複雑なロジックを組み、高品質な記事を作成
   - 一記事一記事のクオリティを高めることに注力

3. **AIが担当すべき領域を取る**
   - 人間の手作業と明確に分離された「AIが担当すべきライティング領域」を確立
   - その領域における必須ツールとして浸透する

---

**成功の定義（5年後）:**

> AI時代において「ブログの自動運用部分はArgo Note」という認識が当たり前になっている状態。
> 人間が書くべき記事と、AIに任せるべき記事が明確に分かれ、後者の領域でデファクトスタンダードとなる。

#### A4. 最も重要な差別化要素

**決定事項:**

**「生成AIによる超自動化エージェントツール」**としてのポジショニング

**詳細:**

1. **バイブコーディングの体験を模倣**
   - バイブコーディングで「簡単にアプリを開発できる」という体験が流行している
   - それに準じた体験として「簡単な入力をするだけで自分のWordPressブログが生成され、自動で運用されていく」を提供
   - 最先端な印象をユーザーに与える訴求軸

2. **全自動の一気通貫体験**
   - ユーザー入力を最小限に抑制
   - WordPressサイトのセットアップ
   - 記事の情報収集
   - 記事テーマの提案
   - 記事の分析
   - ライティング
   - 公開
   - 日々の運用
   - **これらすべてを自動で実行**

3. **競合との差別化方針**
   - AutoBlogging.ai等の競合調査を実施 → **完了**
   - 競合の優れている点は積極的に採用
   - 技術的優位性より「体験の魅力」で勝負

#### A4-2. 競合分析結果（AutoBlogging.ai）

**調査日:** 2026年1月25日

**AutoBlogging.aiの概要:**
- 月間ユーザー数: 40,000+
- 生成記事数: 100万+
- 評価: 4.9★（2026年 SEOコミュニティで#1評価）

**価格体系（クレジット制）:**

| プラン | 月額 | クレジット数 |
|--------|------|------------|
| Starter | $12 | 40記事 |
| Regular | $32 | 120記事 |
| Standard | $64 | 300記事 |
| Gold | $116 | 600記事 |
| Premium | $162 | 1,000記事 |

**主要機能:**
- Quick Mode / Godlike Mode（SERP分析付き）
- Bulk Generation（一括500記事）
- Topical Maps（記事クラスター戦略提案）
- Semantic SEO Analysis（21項目監査）
- WordPress公式プラグイン連携
- Human Proofreader（人間レビュー）オプション

**Argo Noteが採用検討すべき機能:**

| 機能 | 優先度 | 理由 |
|------|--------|------|
| Topical Maps | 高 | 既にC1で「記事クラスター提案」として計画済み |
| SERP分析 | 高 | 競合記事の分析でSEO効果向上 |
| Semantic SEO | 中 | NLPキーワード最適化で検索順位改善 |
| Human Proofreader | 低 | 有料オプションとして後日検討 |

**Argo Noteの明確な差別化ポイント:**

| 項目 | AutoBlogging.ai | Argo Note |
|------|-----------------|-----------|
| WordPressセットアップ | **ユーザー自身** | **完全自動（弊社提供）** |
| サーバー管理 | ユーザー責任 | 弊社が運用 |
| 体験 | ツール提供 | **オールインワン・放置OK** |
| 課金体系 | クレジット制（記事単位） | サブスク（固定月額） |
| ターゲット | SEO経験者向け | **初心者・バイブコーダー向け** |

**結論:**
> AutoBlogging.aiは「SEOライティングツール」、Argo Noteは「自動化されたブログ運用サービス」。
> **ポジショニングが根本的に異なる**ため、直接競合ではなく「代替手段」の関係。
> AutoBlogging.aiを使うには、ユーザーがWordPressを自分で構築・管理する必要がある。
> Argo Noteは「何もしなくていい」体験を提供することで差別化可能。

### B. ターゲット顧客

#### B1-B2. 最優先ターゲット

**決定事項:**

**プライマリターゲット: バイブコーディングでプロダクトを作った個人開発者**

---

**なぜバイブコーダーをターゲットとするのか:**

1. **時代の潮流**
   - 2026年現在、バイブコーディングが爆発的に流行しており、これからさらに拡大する
   - 誰でもアプリケーションを作れる時代が到来

2. **彼らが直面する実態**
   - バイブコーダーが作ったアプリケーションの**ほとんどは利用されない**
   - 原因：集客方法がわからない / 集客チャネルを持っていない / マーケティング知識がない
   - プロダクトは完成しても、誰にも届かない

3. **訴求の刺さりやすさ**
   - 時代の潮流に沿ったターゲット設定により、「**自分のことを言われている**」と即理解できる
   - 抽象的な「ブログ自動化」ではなく、彼らの課題に直接響く訴求が可能

---

**重要：マーケティング手段を持っていないとは**

バイブコーダーは以下の**すべて**に該当する：
- **知識がない** - SEOやブログの書き方がわからない
- **時間がない** - 本業や開発で手一杯
- **興味がない** - マーケティングより開発に集中したい

---

**本サービスが提供する集客チャネル:**

単なる「SEOブログ」ではなく、複合的な集客チャネルとして機能：

| チャネル | 効果 |
|---------|------|
| **SEO（検索上位表示）** | 検索からの継続的な流入 |
| **Google Discover** | プッシュ型の露出、バイラル可能性 |
| **ドメインパワー蓄積** | AI検索（Perplexity等）でも引っかかる基盤 |
| **LP補完** | プロダクト説明・関連情報ページとしての役割 |

---

**対象プロダクトの範囲:**

> **Webからアクセス・購入・ダウンロードするプロダクトであれば、すべてが対象**

- Webアプリ / SaaS
- モバイルアプリ（ストア誘導）
- Chrome拡張機能
- AIツール / ボット
- ノーコードで作ったサービス
- その他、Webに接点があるすべてのプロダクト

**理由:** バイブコーディングで作成可能なもの全てが対象。Webに接点があればブログで集客できる。

---

**段階的ターゲット拡張:**

| フェーズ | ターゲット | 訴求 |
|---------|-----------|------|
| **初期（MVP）** | バイブコーダー | 「作ったプロダクトの集客を自動化」 |
| **成長期** | 自社プロダクト保有者全般 | 「プロダクト集客の自動化」 |
| **拡大期** | ブロガー、アフィリエイター | 「自動で運用できる資産構築」 |

**注意:** 初期は**バイブコーダーに刺さる訴求**に集中。汎用的な「WordPress自動化」訴求は行わない。

---

**本サービスが提供する価値:**

> **半自動的・自動的にブログ運用ができるという魅力**を提供する。
> これは**単なる「WordPress自動化」という機能的価値のみを訴求するサービスではない。**
> 「自分のプロダクトへの集客を、専門知識なしに自動化できる」という**体験価値**を提供する。

**理由:**
- ブログ運用の優先度は開発者にとって最優先ではない
- SNS運用やショート動画制作より優先度が1-3段低い
- だからこそ「自動化で埋められるなら、多少の費用を払ってでも可能性を買う」層が存在

**需要検証方法（Q1回答）:**
- Phase 0（モックアップフェーズ）で検証実施
- ウェイティングリスト収集
- ビルドインパブリック / プロセスエコノミーで需要テスト

**ブログの多面的価値（Q1回答）:**
1. **SEO上位表示** - 検索からの流入
2. **Google Discover** - プッシュ型の露出
3. **LP補完** - ツール説明・関連情報ページとしての役割
4. **ドメインパワー向上** - AI検索（Perplexity等）でも引っかかるための基盤構築

**ポジショニングの差別化:**
> SNS自動化ツールはレッドオーシャン（競合との細部クオリティ比較になる）。
> ブログ運用・SEO集客・ディスカバー集客という「優先度が低い領域」に焦点を当てることで、ポジショニングをずらし、より幅広いターゲットを対象にできる。

#### B12. ユーザーにとっての「成功」の定義

| 中 | **放置できる安心感** | 何もしなくてもブログが更新され続ける |

**注意:** すべてが同時に達成される必要はない。**いずれかの成功体験**があれば継続利用につながる。

#### B13. 期待値コントロール方針

**決定事項:**

| 方針 | 内容 |
|------|------|
| **訴求する価値** | 「自動でコンテンツが蓄積される」こと |
| **訴求しない価値** | 「SEO上位表示を保証する」 |
| **並列で訴求** | SEO以外の価値（Google Discover、ドメインパワー蓄積、AI検索対応） |

**具体的な訴求スタンス:**

1. **「保証」はしない**
   - 「SEOで必ず上位表示される」とは言わない
   - AI生成コンテンツの効果には不確実性があることを前提とする

2. **「蓄積」を価値とする**
   - 放置していてもコンテンツが積み上がっていく
   - Webページという資産が自動で構築される

3. **SEO以外の価値も訴求**
   - Google Discoverでのバイラル可能性
   - ドメインパワーの蓄積（長期的なSEO基盤）
   - AI検索（Perplexity等）での引用可能性

**利用規約への反映:**
- AI生成コンテンツであること、SEO効果を保証しないことを明記

#### B15. 記事クオリティ戦略（MVP vs 将来）

**決定事項:**

> 記事クオリティへの注力度合いは**MVPと将来でグラデーションを持たせる**

**将来の方針:**
- 複数段階のレビュー・推敲プロセスを組み込む
- A6で掲げた「一記事一記事のクオリティにフォーカス」を実現
- AIが担当すべき領域でのクオリティNo.1を目指す

**注意:** A6のビジョン（クオリティ重視）は将来の姿であり、MVPでは達成を求めない。

#### B10. 先行投資型アプローチの戦略的根拠

**本プロジェクトの根本的な開発哲学:**

このプロジェクトは「現在売れるか」ではなく、**「将来必須ツール化される市場に先行参入する」**という戦略に基づいている。

**1. 市場成熟を待ってからでは遅い**

> 将来的にこのようなAIブログ運用ツールが特定のユーザーにとって必須ツール化されることが想定される。
> そのような未来が実現する段階になってから開発を始めたら、**圧倒的に遅い**。

- 市場が成熟してからの参入は「レッドオーシャン」への参入を意味する
- 先行者優位（First Mover Advantage）を確保するためには、市場が未成熟な段階から開発を進める必要がある
- 「売れない状態」であっても、プロダクトを保持し、技術基盤・ノウハウ・ユーザーベースを蓄積しておくことが重要

**2. 地盤を盤石にしておくことの本質的価値**

```
市場成熟前        市場成熟時点        市場爆発期
    │                  │                  │
    ▼                  ▼                  ▼
 [開発・改善]  →  [ユーザー基盤確立]  →  [スケール]
    │                  │                  │
  技術蓄積          信頼構築           収益化
  運用経験          口コミ基盤         市場シェア
  プロンプト最適化   コミュニティ形成    ブランド確立
```

- 市場が爆発する前に「すでに動いているプロダクト」を持っていることが競争優位
- 運用経験から得られるノウハウ（WordPress Multisite運用、AI記事品質改善など）は後発者が簡単に得られない資産
- ユーザーコミュニティが形成されていれば、市場成熟時に一気にスケール可能

**3. 「現時点で効果がない」ことは撤退理由にならない**

- SEO効果が現時点で低くても、LLM技術は急速に進化している
- 2024年のLLMと2026年のLLMでは品質が大きく異なる
- プロンプトエンジニアリング、RAG、Agenticワークフローなどの手法も進化中
- **プロダクトの「器」を先に作り、技術進化に合わせてAI部分を差し替えていく**戦略

**この方針に対するリスク認識:**

| リスク | 対策 |
|--------|------|
| 市場が成熟しない | Phase 0でビルドインパブリック検証（→ B11参照） |
| 資金が尽きる | 月$100の低コスト運用（→ I2参照） |
| 競合が先に市場を取る | 差別化ポイント（オールインワン体験）の強化 |
| 技術が期待通り進化しない | ピボット可能な設計（Exit Strategy確保） |

#### B11. ビルドインパブリック戦略（需要検証・コミュニティ形成）

**開発哲学: プロセスエコノミー × ビルドインパブリック**

本プロジェクトは、従来の「開発 → リリース → マーケティング」という順序ではなく、**開発プロセス自体を公開し、ユーザーと共に作り上げる**アプローチを採用する。

**1. ビルドインパブリック（Build in Public）とは**

> 開発の過程をオープンに発信し、ユーザー・コミュニティからのフィードバックを受けながらプロダクトを改善していく手法。

**実施内容:**
- 開発進捗のリアルタイム発信（X/Threads）
- 設計判断の背景・理由の公開
- 失敗・試行錯誤のプロセスも隠さず共有
- ユーザーからの意見・要望をディスカッション形式で収集

**2. プロセスエコノミー**

> 完成品だけでなく、「作る過程」自体に価値を見出す経済圏。

- ユーザーは「完成品を買う」だけでなく「一緒に作る体験」に価値を感じる
- 開発者とユーザーの距離が近いことで、ロイヤリティ・信頼が構築される
- β版ユーザーは「最初期から応援していた」という帰属意識を持つ

**3. Phase 0における需要検証フロー**

```
Step 1: コンセプト発信
   │  - サービスのビジョン・価値提案を発信
   │  - 「こんなツールがあったら使いたいか？」を問いかけ
   ▼
Step 2: モックアップ・デモ動画公開
   │  - 実際の画面イメージ・操作フローを視覚化
   │  - ユーザーの反応（いいね、RT、コメント）を定量観測
   ▼
Step 3: ウェイトリスト収集
   │  - 「リリース時に通知を受け取りたい人」を募集
   │  - 目標: 50名
   ▼
Step 4: ディスカッション
   │  - 登録者との対話（DM、コミュニティ）
   │  - 「何に最も価値を感じるか」「何が不安か」をヒアリング
   ▼
Step 5: 軌道修正
   │  - フィードバックを基に機能優先度・訴求軸を調整
   │  - 大きなニーズが見つかればピボットも検討
   ▼
Step 6: 本開発開始（Phase 1〜）
```

**4. ビルドインパブリックで得られるもの**

| 得られるもの | 詳細 |
|------------|------|
| **需要の実証** | 反応がなければ早期にピボット可能 |
| **初期ユーザー** | 開発段階から関わったユーザーは継続率が高い |
| **プロダクト改善** | ユーザー視点のフィードバックで的外れな開発を防ぐ |
| **マーケティング効果** | 開発ストーリー自体がコンテンツになる |
| **信頼構築** | 透明性がブランドへの信頼を生む |

**5. 発信チャネルと頻度**

| チャネル | 頻度 | 内容 |
|---------|------|------|
| X（Twitter） | 毎日 | 開発進捗、学び、小ネタ |
| Threads | 毎日 | Xと連動 |
| note/Zenn | 週1-2回 | 技術的な深掘り記事 |
| Product Hunt | リリース時 | ローンチ告知 |
| Indie Hackers | 週1回 | 進捗レポート、議論参加 |

**6. 軌道修正の判断基準**

| 指標 | 良好 | 要検討 | 撤退/ピボット検討 |
|------|------|--------|-----------------|
| ウェイトリスト登録数（2週間） | 50+ | 20-49 | <20 |
| モックアップ動画の反応率 | 5%+ | 2-5% | <2% |
| ディスカッションでの「欲しい」声 | 多数 | 少数 | ほぼなし |
| 「お金を払ってでも使いたい」という声 | あり | 曖昧 | なし |

**7. 「大きなニーズを掘り当てる」プロセス**

> 最初から完璧なプロダクトを作ろうとしない。
> ユーザーとのコミュニケーションの中で、細かく軌道修正しながら、大きなニーズを掘り当てていく。

- 初期仮説: 「個人開発者はAI自動ブログを求めている」
- 検証: ビルドインパブリックでの反応観測
- 発見: 実際のニーズは仮説と異なる可能性がある
- 対応: ニーズに合わせてピボット or 機能調整

**これはスタートアップ的な「リーンスタートアップ」の考え方と整合する:**
- Build（構築）→ Measure（計測）→ Learn（学習）のサイクル
- 最小限のコストで市場検証
- 失敗を早く・安く行い、成功パターンを見つける

### C. 提供機能・UX

#### C2. セットアップ時間

**決定事項:**

> 「3分で完了」は**ただの目安であり根拠なし**。時間ではない。

**本質的に重要なこと:**
- ユーザーの**認知負荷・認知ストレスの低減**
- 「簡易的にできる」ということがクリティカル

#### C3. 記事生成頻度

**決定事項:**

> 「毎日1記事」は**固定ではない。根拠なし。**

**本質的に重要なこと:**
- ユーザーが指定したスケジュールに則って記事を生成
- カスタマイズ性が重要

**例:**
- 1日10記事生成
- 1日1記事生成
- 1日1回、複数記事を更新

#### C1. MVP必須機能

**決定事項:**

| 優先度 | 機能 | 詳細 |
|--------|------|------|
| 1 | 簡易入力 | 認知負荷を下げる簡略化された入力フロー |
| 2 | WordPress自動セットアップ | 弊社サーバーでほぼ自動で完成。バックエンドAPIサーバーとWPサーバーを分離 |
| 3 | 記事クラスター提案 | 簡易ヒアリング/入力から生成AIが記事クラスターを提案し、スケジュールに則って順次生成 |

**MVP縮小の可能性:**
> さらにMVPを最小単位に落とし、「入力した内容に対し、手動で1記事ずつ生成する」レベルまで分解する可能性あり。要検討。

#### C11. 自動化機能の扱い（Q6回答）

**決定事項:**

> MVPリリースまでに**自動化機能（スケジュール生成）を含めるフェーズを追加**する

**理由:**
- 「自動で資産が積み上がる」「放置でOK」という訴求と実機能の乖離を解消
- 訴求を変更するのではなく、機能を訴求に合わせる方針

**アクション:**
- [x] Phase構成を見直し、自動化機能をMVP（Phase 5以前）に含める → **Phase 4に移動完了**

#### 追加機能: 成約導線設計

**決定事項:**

ユーザーが成約させたいLP/商品を設定 → 記事内でその商品への導線を描く

**詳細:**
- Tavily API等によるセマンティック検索で記事クラスター構想を提案
- 各記事の文末、またはピラーページに関連付け
- 成約ページへの誘導箇所を設定で定義
- アフィリエイト商品の成約導線としても機能

### D. 価格・プラン

#### D1. 価格戦略

**決定事項:**

| フェーズ | 価格 | 目的 |
|---------|------|------|
| 初期（拡散期） | **$20/月**（約3,000円） | 拡散・ユーザー獲得優先 |
| 成長期 | **$30/月**（約4,500円） | 収益化強化 |
| 並列 | 高額プラン | 機能拡張版として提供 |

**補足（Q4回答）:**

> 初期フェーズは**利益度外視でバイラルを目指す**。いきなり回収することはせず、可能な限り広く展開していくことを目指す。

**戦略的意図:**
- ユーザーに対して最大限の価値提供を行う
- サービスが良ければ「このツールを使い続けなければならない」という価値観になる
- 大きく回収するのはユーザー数が増えてから
- **回収フェーズを明確に分ける**という考え方

#### D7. フリーミアムモデル（無料プラン）

**決定事項:**

> **機能制限付き + 期間限定の無料プランを設ける**

**無料プランの範囲:**
- ブログのセットアップ完了まで無料
- 数記事の生成・公開まで無料
- 「運用できている状態」まで体験させる

**制限の発動条件:**
- 一定期間経過後、または
- 用意されたクレジット/記事数を使い切った場合

**価格帯の相場観:**
- AIツール一般の相場：$10〜$40程度
- スタンダードプラン：**$20程度**を想定

---

**【検討課題】無料プランの制限方式**

以下の方式を検討中。実装方法は要検討。

| 方式 | 例 | メリット | デメリット |
|------|-----|---------|-----------|
| **記事単位制限** | 3記事まで無料 | シンプル、わかりやすい | LLM壁打ち等の消費が考慮されない |
| **クレジット制** | 800クレジットまで無料 | 全操作を公平に計測 | ユーザーにわかりにくい |
| **期間制限** | 14日間無料 | 明確な期限 | 短期間で価値を感じられない可能性 |
| **ハイブリッド** | 14日間 or 3記事のいずれか早い方 | バランス | 複雑 |

**注意点:**
- LLMとの壁打ち（ブログコンセプト設計等）でクレジットを多く消費した場合、記事生成前に無料枠が尽きる可能性
- ユーザー体験として「運用を体験できた」と感じられる設計が重要

#### E12. SEO戦略駆動型コンテンツ生成フロー

**決定事項: 7フェーズSEO戦略パイプラインを採用**

単純な「URL → クロール → 記事生成」ではなく、SEO戦略に基づいた体系的なコンテンツ生成を行う。

**フロー概要:**

```
Phase A → Phase B → Phase C → Phase D → Phase E → Phase F → Phase G
[プロダクト] [購買思考] [KW調査] [競合分析] [クラスター] [記事生成] [最適化]
```

| Phase | 名称 | 処理内容 |
|-------|------|----------|
| A | プロダクト理解 | URL/対話/競合調査から情報収集 |
| B | 購買思考推論 | LLMで購買ファネルに沿った検索キーワード仮説生成 |
| C | キーワード調査 | API（Keywords Everywhere/DataForSEO）で検証・拡張 |
| D | 競合/SERP分析 | Tavily + LLMで上位記事分析、差別化設計 |
| E | 記事クラスター設計 | ピラー・クラスター構造、内部リンク戦略 |
| F | 個別記事生成 | Planner → Writer → Editor → Illustrator |
| G | パフォーマンス最適化 | GSCデータ連携、リライト提案（Phase 10連携） |

**キーワード調査API選定:**

| API | 月額目安 | 採用優先度 |
|-----|---------|-----------|
| Keywords Everywhere | $10〜 | MVP候補 |
| DataForSEO | $50〜 | 成長期候補 |
| Semrush API | $100〜 | 将来検討 |

**詳細:** [AIパイプライン](./architecture/04_AI_Pipeline.md)

#### H17. ブランディング戦略（二段階アプローチ）

**決定事項: MVP用ブランディングとブランド進化を分離**

| 成果物 | 説明 |
|--------|------|

| カラーパレット | プライマリ・セカンダリ・アクセント色を定義 |
| タイポグラフィ | フォントファミリー選定 |
| OGP画像 | SNSシェア用画像テンプレート |
| ファビコン | ブラウザタブ用アイコン |

| 施策 | 説明 |
|------|------|
| ロゴリファイン | プロデザイナーによる洗練（予算確保後） |
| ブランドガイドライン | 詳細な使用ルールを文書化 |
| アニメーション | ローディング・マイクロインタラクション |
| 拡張アセット | プレゼン資料、名刺、SNSバナー |

**判断基準:**
- Phase 0.5完了時: 「信頼できそうなサービス」という印象を与えられるか
- Phase 13実施時: MRR $1,000達成、またはユーザー100人到達後
